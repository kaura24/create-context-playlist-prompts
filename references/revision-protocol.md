# 승인·수정 프로토콜

## 상태를 유지하라

다음을 대화 안에서 간단히 관리하라.

- 승인된 장르 좌표, 레퍼런스 DNA, 보컬 명단
- 승인된 10곡 전체 설계표
- 곡별로 사용한 실제 보조 레퍼런스와 화성 레퍼런스
- 승인된 곡과 아직 만들지 않은 곡
- 누적된 공통 요구사항

같은 일반 보조 레퍼런스를 여러 곡에 반복 사용하는 것을 허용하라. 최소 3개의 실제 화성 레퍼런스는 허용된 영어 팝송·일본어곡·한국어곡 안에서 선정하라. 화성 레퍼런스의 트랙 간 부분 중복은 허용하지만 두 트랙 간 중복률 50% 이상은 불허하며, 전체 설계표에 중복 사용된 곡의 역할과 적용 범위를 기록하라.

## 초기화와 곡별 재설계를 구분하라

사용자가 `초기화`, `전체 초기화`, `완전 초기화`, `프로젝트 초기화`, `플레이리스트 리셋`을 명시하면 항상 완전 리셋으로 처리하라. 현재 곡을 다시 만드는 요청이나 일부 상태 초기화로 축소하지 마라.

초기화 시 다음 순서로 상태를 전환하라.

1. 현재 승인 상태, 곡별 버전과 피드백을 히스토리에 `초기화 전 보관` 상태로 남기고 삭제하거나 덮어쓰지 마라.
2. 활성 상태의 Context, Primary·Supporting reference, Target·Rejected render, Approved constraints, 장르 좌표, 언어, 보컬 명단과 발성, 금지 조건, 누적 공통 규칙을 전부 비워라.
3. 이전 10곡 설계표와 승인, 현재 곡 선택, 곡별 생성 순서, Prompt·Lyrics 완료 상태를 전부 비워라. 초기화 전 값을 새 프로젝트의 기본값이나 추정값으로 사용하지 마라.
4. 새 프로젝트의 Context와 Primary reference부터 초기 인터뷰를 다시 시작하라. 한 번에 1~2개만 질문하고, 이전 답을 사용자가 새로 명시하지 않는 한 복원하지 마라.
5. 상위 입력과 보컬 승인이 갖춰지면 정확히 10행의 새 전체 설계표를 작성하라. 새 설계의 승인을 요청하고 멈춰라. 승인 전에는 어떤 단일 곡의 Suno Prompt나 Lyrics도 출력하지 마라.

`현재 곡을 처음부터 재설계`, `이 곡을 전면 수정`, `Prompt와 Lyrics를 다시 작성`은 곡별 전체 재생성이다. 이 표현만으로 10곡 설계 승인을 해제하지 마라. `처음부터`라는 말이 프로젝트와 현재 곡 중 어느 범위인지 불명확하고 결과가 크게 달라질 때만 한 번 확인하라.

## 마음에 들지 않는 이유를 집요하게 인터뷰하라

사용자가 변경이나 재생성을 요청하면 바로 다시 만들지 마라. 무엇이 마음에 들지 않는지 충분히 특정할 때까지 한 번에 한두 가지씩 후속 질문하라. 이미 구체적으로 답한 내용을 반복해서 묻지 마라.

필요에 따라 다음 축을 파고들어라.

- 레퍼런스 유사도가 너무 높거나 낮은가?
- 장르 좌표 중 국가, 연대, 하위 장르, 언어, 속도 해석이 틀렸는가?
- 보컬의 음색, 발성, 최저음, 감정 깊이, 숨소리, 억양이 문제인가?
- 악기 구성, 전주, 드럼 연주, 코드 진행, 고유 화성 장치가 문제인가?
- 훅의 기억성, 반복 횟수, 멜로디 밀도, 곡 구조가 문제인가?
- 가사의 주제, 관점, 서사, 언어, 표현 강도, 음절 밀도가 문제인가?
- 길이, 에너지, 믹싱, 공간감, 청취 상황 적합성이 문제인가?
- 다른 곡과 너무 비슷하거나 플레이리스트 세계관에서 벗어났는가?

“별로다”, “다르게”, “더 좋게”처럼 모호한 답이면 구체적인 청감과 원하는 반대 방향을 물어라. 사용자가 충분히 구체적인 수정 지시를 주면 인터뷰를 종료하고 재생성하라. 사용자가 즉시 재생성을 명시하면 확인된 정보만으로 진행하라.

## 곡별 수정과 공통 수정을 구분하라

현재 곡에만 해당하면 곡별 수정으로 처리하라. 현재 곡의 통합 Suno Prompt와 Lyrics를 함께 재설계하고 다른 곡은 바꾸지 마라.

현재 곡과 앞으로 만들 모든 곡에 해당하면 공통 수정으로 처리하라. 누적 공통 요구사항에 추가하고 현재 곡을 재생성하라. 남은 곡의 설계에도 적용하되 이미 승인된 곡은 사용자가 요청하지 않는 한 바꾸지 마라.

구분이 애매하고 결과가 크게 달라질 때만 현재 곡 전용인지 공통인지 확인하라.

## 재생성하라

1. 확인된 불만과 목표를 한국어로 짧게 재진술하라.
2. 곡별 또는 공통 수정으로 분류하라.
3. 기존 프롬프트를 편집 원본으로 사용하지 말고 승인된 전체 설계와 최신 요구사항에서 스타일과 가사를 처음부터 다시 판단하라.
4. 실패 표현, 폐기 조건, 상충·중복 문구를 제거한 통합 Suno Prompt와 Lyrics 전체를 완전 리팩터링하여 재출력하라. 부분 문구, 차이점, 추가 지시만 제공하지 마라.
5. 배제 조건이 통합된 Suno Prompt 800자, 긍정·부정 지시 무충돌, 단일 가사 언어, 장르 표준 시간 ±20%, 보컬 일관성, 중복 레퍼런스의 역할·적용 범위, 화성 레퍼런스 중복률 50% 미만을 검사하라.
6. 공통 수정이면 다음 곡에 적용할 누적 규칙을 한 줄로 알리라.
7. 수정 사유, 분류, 새 검사 결과, 새 통합 프롬프트를 해당 `history/YYYY-MM-DD__topic-slug.md`에 새 버전으로 추가하고 루트 `PROJECT_HISTORY.md` 인덱스를 유지하라.

모든 수정에서 현재 곡의 확정 화성 레퍼런스 최소 3개를 다른 모든 곡의 확정·예정 목록과 쌍별 대조하라. 화성 자체를 수정하지 않았더라도 검사를 생략하지 마라. `교집합 수 ÷ 두 목록 중 더 짧은 목록의 곡 수`가 50% 미만이면 중복을 허용하고, 50% 이상이면 충돌을 만드는 항목만 교체한 뒤 다시 계산하라. 반복 사용되는 곡마다 현재 곡에서 참고하는 화성 요소와 다른 트랙에서의 역할 차이를 기록하라.

부분 조각만 수정해 서로 어긋난 결과를 만들지 마라. 모든 수정은 전체 리팩터링 버전으로 제공하라. 사용자가 승인할 때까지 같은 곡을 개선하고 승인 뒤에만 다음 곡으로 이동하라.
